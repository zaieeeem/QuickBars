Store artwork lives in [`assets/play/`](../assets/play/) and is regenerated by

## Store copy

**Short description (≤80 chars):**

> Home Assistant quick bars over any app on your TV. Tap the bell — control home.

**Full description:**

> Bing-Bong puts your Home Assistant controls on top of whatever you're watching.
>
> Press a remote-control button you choose and a quick bar slides in over any app:
> lights, switches, scenes, media players, climate, covers, cameras and more —
> all D-pad friendly and readable from the couch.
>
> ★ Quick bars over any app — no need to switch away from your show
> ★ Direct D-pad control — hold left/right to dim a light or change media volume
> ★ Trigger keys — map single, double or long presses of remote buttons to actions
> ★ Real-time — talks to your Home Assistant over your local network (WebSocket)
> ★ Camera picture-in-picture — glance at the doorbell without leaving your movie
> ★ TV-optimized notifications from Home Assistant, with images and actions
>
> Bing-Bong connects directly to your own Home Assistant on your local network.
> No cloud account, no tracking, no data collection.
>
> Bing-Bong is free software (GPL-3.0), based on QuickBars by Omri Peretz
> (github.com/Trooped/QuickBars). Full source code: github.com/zaieeeem/QuickBars

## Play Console permission declarations

### `SYSTEM_ALERT_WINDOW` (Display over other apps) — core feature

Declaration text:

> Bing-Bong's core and only purpose is a remote-control-summoned overlay for Home
> Assistant. When the user presses a trigger key they configured, the app draws a
> quick-action bar (or camera picture-in-picture) over the currently visible app so
> they can control smart-home devices without leaving full-screen content. The
> overlay appears only on explicit user action (a configured remote key press or a
> notification from the user's own Home Assistant server), is dismissed by BACK, and
> is never used for ads, phishing, or obscuring other apps' UI. Without
> SYSTEM_ALERT_WINDOW the app has no function.

### Accessibility Service API declaration

The `QuickBarService` accessibility service (`isAccessibilityTool="false"`) is used
solely to observe configured hardware key presses (remote-control buttons) so quick
bars can be summoned from any screen. Declaration/prominent-disclosure text is
maintained in-app as `R.string.play_store_accessibility_disclosure` and shown to the
user before enabling the service:

> This app uses the Accessibility Service API to provide its core functionality.
> The Accessibility Service is required to detect and intercept specific hardware
> key presses (e.g., from your remote control) that you configure as "trigger keys."
> This allows the app to launch your quick bars or control your Home Assistant
> devices even when the app is in the background. Bing-Bong does not use the
> Accessibility Service to collect any user data, see your personal information, or
> read the content of your screen. It is used solely for the key press trigger
> feature.

### Foreground service (`FOREGROUND_SERVICE_DATA_SYNC`)

> The optional "persistent background connection" setting keeps a WebSocket to the
> user's own Home Assistant server alive (with a visible persistent notification) so
> Home-Assistant-initiated triggers and TV notifications arrive while other apps are
> in the foreground. It syncs entity state data only; the user enables and disables
> it explicitly in settings, and `RECEIVE_BOOT_COMPLETED` only restores it after
> reboot when that setting is on.

### Data safety form

- No data collected, no data shared. All communication is device ↔ user's own Home
  Assistant server on the local network (`usesCleartextTraffic="true"` because HA
  installs commonly run plain HTTP on the LAN; note this in the form's security
  section). The HA access token is stored encrypted on-device (Tink/EncryptedPrefs).
- In-app purchases: the manifest declares `com.android.vending.BILLING` and ships a
  RevenueCat-backed "Plus" upgrade screen. **Decision needed before release:** either
  configure Play billing products + a RevenueCat key for the new package, or strip
  the billing permission/upgrade flow. Don't ship a dead purchase button.

## Publication checklist

1. [ ] **GPL source availability (BLOCKER):** make `zaieeeem/QuickBars` public (or
       publish a public mirror and link it from the listing + About screen) *before*
       the first binary goes to Play. GPLv3 §6 — binaries must come with source.
2. [ ] **Signing:** generate a NEW private upload keystore. The committed
       `.ci/quickbars-ci.jks` (passwords in the workflow file) is public and must
       never sign a Play build. Store the new keystore + passwords in the secrets
       manager (Bitwarden/Infisical) per AGENTS.md §6/§8; enroll in Play App Signing.
3. [ ] **Billing decision:** wire RevenueCat/Play products for `com.zaiemv.bingbong`
       or remove the Plus/upgrade flow + `BILLING` permission.
4. [ ] Create the app in Play Console (name **Bing-Bong**, id `com.zaiemv.bingbong`),
       opt in to **Android TV** release track; first AAB upload is manual
       (AGENTS.md §8), then CI publishes via the Play API to internal testing.
5. [ ] Build an **AAB** for Play (`:app:bundleRelease`) — the GitHub release APK
       workflow stays for Obtainium sideloads.
6. [ ] Upload listing assets from `assets/play/`:
       - [ ] Hi-res icon 512×512 (`icon-512.png`)
       - [ ] **TV banner 1280×720** (`tv-banner-1280x720.png`) — required for TV listings
       - [ ] Feature graphic 1024×500 (`feature-graphic-1024x500.png`)
       - [ ] ≥2 TV screenshots (16:9, ≥1920×1080) — capture from a device/emulator (TODO)
7. [ ] Paste store copy (above) + privacy policy URL.
8. [ ] Fill declarations: SYSTEM_ALERT_WINDOW is not a Play declaration form, but the
       **Accessibility API** declaration and **Foreground service** declaration are —
       use the texts above. Complete the **Data safety** form (no collection).
9. [ ] Content rating questionnaire (utility, no UGC → Everyone).
10. [ ] TV quality review: Google manually reviews TV apps against the
        [TV app quality guidelines](https://developer.android.com/docs/quality-guidelines/tv-app-quality)
        — D-pad-only operation, banner, no touchscreen requirement (all verified above).
11. [ ] After approval: promote internal → production when ready.
